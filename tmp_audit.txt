#### **2. `advance_workflow(workflow_run_id)` ƒ?" Line 372**

**Behavior**:
- Fetches all tasks for workflow
- Identifies `completed_ids` (tasks with status='completed')
- For each PENDING task: checks if all `depends_on` tasks are in `completed_ids`
- If ready: marks task as RUNNING and returns it

**Persistence**: ƒo" PostgreSQL (task status UPDATE)

**Retry Logic**: ƒ?O NONE
- Tasks are NOT retried on failure
- Once marked FAILED, they stay FAILED

**Issues**:
- Task state transitions are not atomic; no lock prevents concurrent advancement
- If `advance_workflow()` called twice in parallel, both may mark the same task as RUNNING

#### **3. `complete_task(task_id, result_data)` ƒ?" Line 537**

**Behavior**:
- Marks task as COMPLETED with output_data = result_data
- **Does NOT auto-call `advance_workflow()`**
- Caller must call `advance_workflow()` separately to trigger next task

**Persistence**: ƒo" PostgreSQL (task status UPDATE)

**Critical Issue**: ƒ?O **NO AUTO-ADVANCEMENT**

- Example flow:
  ```

  2. backend_code must call advance_workflow(workflow_run_id)
  3. advance_workflow() then marks budget_verification as RUNNING
  4. backend_code must invoke budget_verification agent
  ```

- **Risk**: If backend code forgets step 2, workflow hangs forever in RUNNING state

- **Evidence**: `orchestrator.py:1371` calls `advance_workflow(workflow_run_id)` explicitly at startup, but NOT after each task completion; relies on orchestrator's sequential execution to advance

### **SSE Stream Lifecycle**

**Flow**:
1. Client POST `/api/agentic/execute/stream`
2. Server creates EventStream object
3. Server emits RECEIVED, CLASSIFYING, ROUTING events (immediate)
4. Server invokes agents (blocking async/await)
5. Each agent may emit events via event_stream
6. Server emits final COMPLETE or ERROR
7. Stream closes

**Issue**: ƒ?O **STREAM DOESN'T PAUSE AT HUMAN GATES**

- When workflow reaches human_action_required, orchestrator returns P2P response with gate metadata
- Event stream emits COMPLETE event with full gate in payload
- Stream closes
- **Frontend must re-open new SSE stream to `/p2p/resume` to continue**

**Risk**: If frontend crashes between gate and resume, workflow hangs in database forever

### **Gate Reconnection**

**No built-in reconnection mechanism**:
- Once stream closes at gate, client loses connection ID
- Must call `/api/agentic/p2p/resume` endpoint separately
- No way to re-subscribe to existing workflow's event stream

**Workaround**: Polling `/api/p2p/journey/{workflow_run_id}` to check status
